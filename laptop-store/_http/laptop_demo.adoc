= Пример работы интернет-магазина, разбитого на микросервисы

В данном примере интернет-магазин состоит из нескольких микросервисов, каждый из которых отвечает за свою часть бизнес-логики. Ниже приведён список сервисов, а также последовательность запросов (`curl`), демонстрирующих полный цикл обработки заказа: от поиска товара до его доставки (и потенциальной отмены, если потребуется).

== Архитектура и взаимодействие сервисов

1. **Catalog Service (Сервис каталога)**
- Отвечает за хранение и выдачу информации о товарах.
- Предоставляет методы поиска товаров по критериям и получения подробной информации о конкретном товаре.

2. **Inventory Service (Сервис склада)**
- Управляет остатками товаров на складе.
- Позволяет узнать текущую доступность товара и зарезервировать нужное количество единиц под заказ.

3. **Ordering Service (Сервис заказов)**
- Управляет жизненным циклом заказа: от момента создания (PLACED) до подтверждения (CONFIRMED).
- После подтверждения инициирует процесс оплаты и доставки.

4. **Order Tracking Service (Сервис отслеживания заказов)**
- Хранит сводную информацию о статусе заказа, платеже и доставке.
- Позволяет клиентам одним запросом увидеть общее состояние (OrderStatus, PaymentStatus, ShippingStatus).

5. **Payment Service (Сервис оплаты)**
- Принимает запрос на оплату заказа.
- Меняет состояние оплаты (IN_PROCESS → PAID), возможно с некоторой задержкой (например, 30 секунд).

6. **Shipping Service (Сервис доставки)**
- Управляет логикой доставки (старт доставки, отмена доставки).
- Держит свой собственный статус доставки (UNKNOWN, IN_DELIVERY и т.д.).

7. **Invoice Service (Сервис выставления счетов)**
- Формирует счёт (invoice) по заказу.
- Для автоматического создания счёта необходимо запустить Kafka и включить публикацию сообщений в `ordering-service` (настройка `app.disable-message-publishing: false`).

== 1. Catalog Service – поиск и просмотр товара

[source,bash]
----
curl -X GET "http://localhost:8081/catalog?manufacturer=Apple&ramCapacity=8&storageType=ssd"
----

**Ответ**:
----
[
  {
    "model": "MacBook Pro",
    "manufacturer": "Apple",
    "screenSize": 13,
    "screenResolution": "2560x1600",
    "ramCapacity": 8,
    "processorLine": "M1",
    "processorCoreCount": 8,
    "storageType": "SSD",
    "driveCapacity": 256,
    "os": "macOS"
  }
]
----

Получение информации по конкретному ID:

[source,bash]
----
curl -X GET "http://localhost:8081/catalog/1"
----

**Ответ**:
----
{
  "model": "MacBook Pro",
  "manufacturer": "Apple",
  "screenSize": 13,
  "screenResolution": "2560x1600",
  "ramCapacity": 8,
  "processorLine": "M1",
  "processorCoreCount": 8,
  "storageType": "SSD",
  "driveCapacity": 256,
  "os": "macOS"
}
----

== 2. Inventory Service – проверка и резервирование товара

Используется, чтобы:

- Убедиться, что товар действительно есть на складе.
- Зарезервировать нужное количество, чтобы никто другой его не выкупил параллельно.

Проверка наличия товара:

[source,bash]
----
curl -X GET "http://localhost:8082/inventory/byProduct/1"
----

**Ответ**: массив объектов с информацией о складе, где `availableQty` показывает доступное количество.
----
[
  {
    "inventoryId": 1,
    "productId": 1,
    "availableQty": 9,
    "unitCost": 1300
  }
]
----

Резервирование товара:

[source,bash]
----
curl -X POST "http://localhost:8082/inventory" \
     -H "Content-Type: application/json" \
     -d '{
           "inventoryItemId": 1,
           "requiredQty": 1
         }'
----

**Ответ**:
----
{
  "recordId": 1
}
----

== 3. Ordering Service – оформление заказа

Используется, чтобы:

- Сформировать заказ с указанием покупателя, списка товаров и цен.
- Получить уникальный идентификатор заказа, по которому далее можно отслеживать статус.

Создание заказа:

[source,bash]
----
curl -X POST "http://localhost:8085/orders" \
     -H "Content-Type: application/json" \
     -d '{
           "customerId": 1,
           "items": [
             {
               "productId": 1,
               "cost": 3500,
               "qty": 1
             },
             {
               "productId": 2,
               "cost": 2700,
               "qty": 1
             }
           ]
         }'

----

**Ответ**:
----
{
  "id": 1,
  "status": "PLACED"
}
----
Где `id` — это уникальный номер заказа, `status` — начальное состояние (PLACED).

Проверка статуса заказа:

[source,bash]
----
curl -X GET "http://localhost:8085/orders/1/status"
----

(Заказ ещё не оплачен и не подтверждён.)

=== 4. Payment Service – оплата заказа

Используется, чтобы:

- Перевести заказ из состояния неоплаченного в оплаченный (IN_PROCESS → PAID).
- В реальном мире оплата может занимать время (проверка карты, транзакций).

Инициирование платежа:

[source,bash]
----
curl -X POST "http://localhost:8086/payments/byOrder/1"
----
**Ответ**: возвращается идентификатор платежа (например, `2`).

Проверка статуса платежа:

[source,bash]
----
curl -X GET "http://localhost:8086/payments/byOrder/1/status"
----

Через ~30 секунд повторяем запрос и получаем:

[source,bash]
----
curl -X GET "http://localhost:8086/payments/byOrder/1/status"
----

Это означает, что деньги списаны, заказ оплачен.

=== 5. Shipping Service – запуск доставки

Используется, чтобы:

- Организовать доставку оплаченного заказа.
- Можно проверить статус доставки или отменить при необходимости.

Запуск доставки:

[source,bash]
----
curl -X PUT "http://localhost:8087/shipping/1"
----
**Ответ**: возвращается идентификатор доставки (например, `2`).

Проверка статуса доставки:

[source,bash]
----
curl -X GET "http://localhost:8087/shipping/1/status"
----
**Ответ**: "IN_DELIVERY" (или "UNKNOWN", если ещё не обновился статус, или "DELIVERED", если доставка завершена).

Отмена доставки:

[source,bash]
----
curl -X DELETE "http://localhost:8087/shipping/1"
----
**Ответ**: успешное выполнение, если доставка ещё не завершена.

=== 6. Invoice Service – получение счёта

Используется, чтобы:

- Получить чек-фактуру о покупке, необходимый для отчётности, бухгалтерии, печати и т.д.

[source,bash]
----
curl -X GET "http://localhost:8083/invoices/byOrder/1"
----
**Ответ**: данные счёта (invoice).

**Примечание**: Чтобы счёт генерировался автоматически, нужно включить Kafka и включить публикацию сообщений в `ordering-service` (
`app.disable-message-publishing: false` в `application.yml`).

== Итоговый процесс

1. *Найти товар* в каталоге через Catalog Service.
2. *Уточнить остатки* и *зарезервировать* товар через Inventory Service.
3. *Создать заказ* через Ordering Service.
4. *Оплатить заказ* в Payment Service (ждём статус PAID).
5. *Запустить доставку* в Shipping Service (статус IN_DELIVERY).
6. *Получить счёт* (если нужно) через Invoice Service.
7. *Отслеживать сводку* по заказу (статус заказа, оплаты, доставки) можно через Order Tracking Service.

Благодаря микросервисной архитектуре каждый сервис может развиваться независимо, иметь свою базу данных и масштабироваться под свою нагрузку (например, каталог может обслуживать много запросов на поиск, а оплата – меньше, но более тяжёлых транзакций). Это упрощает поддержку и расширение системы.
