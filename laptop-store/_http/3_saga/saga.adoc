= Управление отменой доставки заказа с использованием Saga

В этом документе описывается реализация **паттерна Saga** для управления жизненным циклом заказа в **микросервисной архитектуре** интернет-магазина. **Saga** гарантирует, что если один из этапов выполнения заказа **отменяется**, система автоматически выполняет **компенсационные действия** (например, возврат оплаты при отмене доставки).

Документ охватывает:
- **Что такое Saga и зачем она нужна**
- **Архитектура и взаимодействие сервисов**
- **Пошаговый сценарий обработки отмены доставки**
- **Компенсационный механизм (автоматический возврат средств)**

== Что такое Saga и зачем она нужна?

**Saga** – это шаблон управления распределёнными транзакциями в микросервисной архитектуре.
Когда операция включает **несколько микросервисов**, которые выполняют **цепочку действий**, возникает **проблема согласованности данных**.

Пример:

- Заказ был оформлен и оплачен.
- Началась доставка, но покупатель **отменил заказ**.
- Нужно не только **остановить доставку**, но и **вернуть деньги покупателю**.

**Saga** решает эту проблему, гарантируя, что при отмене процесса будут выполнены **компенсационные действия**.

== Архитектура и взаимодействие сервисов

Основные микросервисы, участвующие в процессе:

1. **Ordering Service (Сервис заказов)**
- Управляет статусом заказа.
- Автоматически инициирует оплату и доставку после подтверждения.

2. **Payment Service (Сервис оплаты)**
- Обрабатывает платёж и следит за его статусом (`IN_PROCESS → PAID`).
- Если доставка отменяется, запускает **возврат денег (refund)**.

3. **Shipping Service (Сервис доставки)**
- Запускает доставку оплаченного заказа.
- Может отменить доставку, вызвав **компенсационное действие в Payment Service**.

4. **Order Tracking Service (Сервис отслеживания заказов)**
- Предоставляет сводный статус (`CONFIRMED`, `PAID`, `IN_DELIVERY`).

== Пошаговый сценарий: отмена доставки заказа

=== 1. Размещение заказа

Создаём заказ с несколькими товарами:

[source,bash]
----
curl -X POST "http://localhost:8085/orders" \
     -H "Content-Type: application/json" \
     -d '{
           "customerId": 1,
           "items": [
             {
               "productId": 1,
               "cost": 2000,
               "qty": 1
             },
             {
               "productId": 2,
               "cost": 1500,
               "qty": 2
             }
           ]
         }'
----

**Ответ**:
----
{
  "id": 1,
  "status": "PLACED"
}
----

=== 2. Подтверждение заказа

[source,bash]
----
curl -X PUT "http://localhost:8085/orders/1/verification"
----

**Ответ**:
----
{
  "id": 1,
  "status": "CONFIRMED"
}
----

=== 3. Оплата заказа

[source,bash]
----
curl -X POST "http://localhost:8086/payments/byOrder/1"
----

**Ответ**:
----
"paymentId": 2
----

Проверяем статус платежа:

[source,bash]
----
curl -X GET "http://localhost:8086/payments/byOrder/1/status"
----

Сначала:
----
"IN_PROCESS"
----

Спустя 30 секунд:
----
"PAID"
----

=== 4. Автоматический запуск доставки

Как только заказ **оплачен**, сервис заказов **автоматически инициирует доставку**:

[source,bash]
----
curl -X PUT "http://localhost:8087/shipping/1"
----

**Ответ**:
----
"deliveryId": 3
----

Проверяем статус доставки:

[source,bash]
----
curl -X GET "http://localhost:8087/shipping/1/status"
----

**Ответ**:
----
"IN_DELIVERY"
----

=== 5. Отслеживание состояния заказа

Можно запросить **общий статус** заказа:

[source,bash]
----
curl -X GET "http://localhost:8084/1"
----

**Ответ**:
----
{
  "orderStatus": "CONFIRMED",
  "paymentStatus": "PAID",
  "shippingStatus": "IN_DELIVERY"
}
----

=== 6. Отмена доставки

Теперь покупатель решает **отменить заказ**.
Мы должны **остановить доставку и вернуть деньги**.

[source,bash]
----
curl -X DELETE "http://localhost:8087/shipping/1"
----

**Ответ**:
----
"CANCELLED"
----

=== 7. Компенсационное действие: возврат оплаты

После отмены **Shipping Service** автоматически уведомляет **Payment Service** о необходимости возврата денег.

[source,bash]
----
curl -X POST "http://localhost:8086/payments/refund/1"
----

**Ответ**:
----
"REFUNDED"
----

=== 8. Логирование возврата денег

Payment Service записывает в логи:
----
"Payment refunded for orderId=1"
----
(В реальном приложении это может быть отмена банковской транзакции и отправка уведомления покупателю).

Теперь заказ отменён, доставка остановлена, а деньги **возвращены покупателю**.

== Итоги

Saga-orchestrated workflow **гарантирует согласованность данных**:

1. Если **всё проходит успешно** → заказ переходит в `IN_DELIVERY`, клиент получает товар.
2. Если **что-то идёт не так** (например, доставка отменена) → запускается **компенсационный процесс** (возврат денег).
3. **Payment Service автоматически делает refund**, когда **Shipping Service отправляет сигнал о отмене**.

Этот подход помогает **избежать несогласованных состояний** (например, оплаченного, но недоставленного товара).

Saga **упрощает управление распределёнными транзакциями**, гарантируя, что **все операции либо выполняются, либо отменяются с компенсацией**.
