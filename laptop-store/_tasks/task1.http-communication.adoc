= Задание 1. Синхронная коммуникация

== Реализация интеграции order-tracking-service и shipping-service

Наш сервис отслеживания заказов (`order-tracking-service`) предоставляет REST-эндпоинт для запроса текущего состояния заказа.
В настоящее время **возвращаемый DTO содержит неполные данные**, поскольку информация о статусе доставки всегда имеет значение `"UNKNOWN"`.

[source,java]
----
@Getter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class OrderTrackingInfo {
	OrderStatus orderStatus;
	PaymentStatus paymentStatus;

	public String getShippingStatus() {
		return "UNKNOWN";
	}
}
----

Чтобы исправить это, необходимо **реализовать интеграцию между `order-tracking-service` и `shipping-service`**,
чтобы клиент получал **фактический статус доставки** вместо заглушечного значения.

Для этого мы будем использовать **Feign-клиент для взаимодействия с `shipping-service`** через REST API.

Вот как это будет выглядеть:

[plantuml,plantuml-diagram,svg]
----

participant "OrderingServiceClient" as OrderingService
database "Order DB" as OrderDB

----

Здесь **`OrderTrackingController`** запрашивает статус заказа у `OrderingServiceClient`, `PaymentServiceClient` и `ShippingServiceClient`. Получение статуса у `OrderingServiceClient` и `PaymentServiceClient` уже реализовано.

Нам осталось добавить интеграцию с `ShippingServiceClient` для получения статуса доставки.

1. **Добавьте зависимость `shipping-shared-model` в `order-tracking-shared-model`**.

4. **Добавьте интеграцию в `OrderTrackingController`**.

=== Шаг 4. Добавьте интеграцию в OrderTrackingController

Добавьте в конструктор `OrderTrackingController` зависимость `ShippingServiceClient`:

[source,java]
.OrderTrackingController.java
----
public OrderTrackingController(OrderingServiceClient orderingServiceClient,
                               PaymentServiceClient paymentServiceClient,
                               ShippingServiceClient shippingServiceClient) {
    this.orderingServiceClient = orderingServiceClient;
    this.paymentServiceClient = paymentServiceClient;
    this.shippingServiceClient = shippingServiceClient;
}
----

Используйте новый клиент для заполнения `OrderTrackingInfo`:

[source,java]
.OrderTrackingController.java
----
public OrderTrackingInfo getOrderStatus(Long orderId) {
    try {
        OrderInfo orderInfo = orderingServiceClient.getOrderStatus(orderId);

        PaymentStatus paymentStatus = null;
        if (orderInfo.getStatus() == OrderStatus.CONFIRMED ||
            orderInfo.getStatus() == OrderStatus.CANCELED
        ) {
            paymentStatus = paymentServiceClient.getStatusByOrder(orderId);
        }

        ShippingStatus shippingStatus =                     // +++
          shippingServiceClient.getShippingStatus(orderId); // +++

        return OrderTrackingInfo.builder()
            .orderStatus(orderInfo.getStatus())
            .paymentStatus(paymentStatus)
            .shippingStatus(shippingStatus) // +++
            .build();

    } catch (Exception e) {
        throw new OrderTrackingException("Error in tracking order",
            orderId, e);
    }
}
----

На этом реализация интеграции завершена.

- Для этого отправьте запрос `PUT /orders/{{orderId}}/verification`

- Используйте запрос `GET /tracking/{{orderId}}`

- Для этого отправьте запрос `PUT /shipping/{{orderId}}`
- Поле `shippingStatus` должно измениться на `IN_DELIVERY`
- Если статус не изменился, у вас что-то пошло не так. Проверьте реализацию каждого шага.

5. **Проверьте статус через 1 минуту**
- Поле `shippingStatus` должно измениться на `DELIVERED`

== Дополнительное задание: реализация верификации заказа

В настоящее время **процесс верификации в `Ordering Service` реализован только на уровне смены статуса заказа**.
Однако **реальная верификация отсутствует** – система **не проверяет наличие товаров на складе** перед подтверждением заказа.

=== Задача
Добавить **настоящий процесс верификации**, который:

- **Запрашивает информацию о наличии товаров в `Inventory Service`**
- **Если товары есть в наличии** → резервирует их **и меняет статус заказа на `CONFIRMED`**
- **Если товаров нет** → меняет статус заказа на `REJECTED` (новый статус)

=== Шаги реализации
1. Добавьте **интеграцию `Ordering Service` с `Inventory Service`** через Feign клиент.
2. При верификации заказа:
- Отправьте **запрос на резервирование товара** в `Inventory Service`.
- Если резервирование успешно → установите статус заказа в `CONFIRMED`.
- Если резервирование не удалось → установите новый статус `REJECTED`.
3. Обновите `OrderTrackingService`, чтобы он **правильно обрабатывал статус `REJECTED`**.

Теперь заказ будет **подтверждён только при наличии товаров на складе**.

Если товаров нет – заказ **будет отклонён автоматически**.

=== Итог

После реализации:

- `OrderingService` **реально проверяет наличие товара** перед подтверждением заказа.
- Интеграция **с `Shipping Service` теперь полностью рабочая**.

Теперь процесс заказа **стал более надёжным, а интеграция завершена**!
