= Выделение доменных объектов в интернет-магазине

== Введение
Выделение **доменных объектов** — один из ключевых шагов при анализе предметной области. Мы можем использовать **Domain Storytelling** и **Event Storming** для идентификации ключевых сущностей, value objects и агрегатов.

== Основные принципы выделения доменных объектов
1. **Анализ бизнес-процессов** → Какие объекты участвуют в сценариях?
2. **Поиск устойчивых понятий** → Объекты, которые имеют бизнес-значение и сохраняют идентичность.
3. **Выделение границ агрегатов** → Что должно управляться как единое целое?
4. **Разделение на сущности и value objects** → Какие объекты имеют идентификатор, а какие — просто атрибуты?
5. **Определение ограниченных контекстов (Bounded Contexts)** → Где заканчивается область ответственности объекта?

== 1. Анализируем Domain Storytelling
В **Domain Storytelling** мы видим, какие объекты упоминаются в процессе работы пользователя. Например:

- **Клиент** добавляет **Ноутбук** в **Корзину**.
- **Система заказов** создаёт **Заказ**, проверяет **Склад**, инициирует **Оплату**.
- **Сервис доставки** создаёт **Отгрузку** и обновляет **Статус заказа**.

*Вывод:* **Ноутбук, Корзина, Заказ, Склад, Оплата, Доставка** — кандидаты в доменные объекты.

== 2. Анализируем Event Storming
Теперь мы смотрим на **события**, в которых эти объекты участвуют:

- `OrderPlaced` → `OrderConfirmed` → `PaymentProcessed` → `ShipmentCreated`
- `StockChecked` → `StockReserved`
- `ItemAddedToCart` → `CartCheckedOut`

*Вывод:* Объекты, которые участвуют в событиях и несут бизнес-логику, являются **доменными объектами**.

== 3. Выделяем сущности и value objects
Теперь классифицируем объекты на **сущности** и **value objects**:

[options="header"]
|===
| **Доменный объект** | **Сущность / Value Object?** | **Описание**

| **Ноутбук (Laptop)**
| Value Object
| Не меняет состояние, нужен только в виде данных

| **Корзина (ShoppingCart)**
| Сущность (Aggregate Root)
| Может изменяться, управляет товарами в корзине

| **Товар в корзине (CartItem)**
| Сущность (часть агрегата)
| Зависит от корзины, хранит количество

| **Заказ (Order)**
| Сущность (Aggregate Root)
| Имеет идентичность, проходит несколько стадий

| **Позиция заказа (OrderItem)**
| Сущность (часть агрегата)
| Привязана к заказу

| **Складская позиция (InventoryItem)**
| Сущность
| Управляет доступностью товара

| **Оплата (Payment)**
| Сущность
| Имеет статус и привязанную сумму

| **Отгрузка (Shipment)**
| Сущность
| Хранит информацию о доставке
|===

*Вывод:* **Сущности** (Order, Payment, ShoppingCart) имеют уникальный ID и изменяются. **Value Objects** (Laptop) — неизменяемые и передаются как данные.

== 4. Определяем агрегаты
Некоторые сущности образуют **агрегаты** — единицы управления данными:

- **Корзина (`ShoppingCart`)** → Включает **товары (`CartItem`)**.
- **Заказ (`Order`)** → Включает **позиции заказа (`OrderItem`)**.
- **Склад (`Inventory`)** → Включает **резервы (`StockReservation`)**.

**Изменения в DDD всегда происходят через агрегат**, например:

- Добавление товара в корзину → `ShoppingCart.addItem(productId, qty)`
- Подтверждение заказа → `Order.confirm()`
- Оплата заказа → `Payment.process()`

_Однако, в Java изменения выполняются через *сервисы*, которые взаимодействуют с агрегатами, а не напрямую через методы агрегатов._

=== Почему сервисы в Java важны?

Потому что сервисы обеспечивают:

-	*Транзакционность* (например, с помощью Spring @Transactional)
-	*Управление жизненным циклом агрегатов* (например, загрузка и сохранение через ORM, Hibernate)
-	*Чистоту модели* (агрегаты остаются “глупыми” и не перегружаются инфраструктурными задачами)

Если бы мы вызывали методы агрегатов напрямую, то не смогли бы обеспечить атомарность операций, а также столкнулись бы с проблемами в работе с Hibernate (например, Lazy Loading или каскадным обновлением).

В других языках программирования можно выполнять изменения напрямую через агрегаты, но в Java это не рекомендуется. Например, в C# можно использовать *Unit of Work* и *Entity Framework* для управления транзакциями и жизненным циклом агрегатов. В Ruby on Rails есть *Active Record*. В Python — *Django ORM*. В PHP — *Doctrine*.

Если говорить о **чистом DDD**, то **изменение агрегатов напрямую** реально возможно в C#, Ruby, PHP и Python. В Java, Go, Kotlin и Scala лучше использовать сервисы и управлять транзакциями вручную.

== 5. Распределение доменных объектов по сервисам

Теперь определим, какие доменные объекты будут входить в каждый сервис:

[options="header"]
|===
| **Сервис** | **Доменные объекты**

| **Каталог (Catalog Service)**
| Laptop

| **Корзина (Shopping Cart Service)**
| ShoppingCart, CartItem

| **Система заказов (Ordering Service)**
| Order, OrderItem

| **Склад (Inventory Service)**
| InventoryItem, StockReservation

| **Оплата (Payment Service)**
| Payment

| **Доставка (Shipping Service)**
| Shipment
|===

Каждый сервис отвечает за управление своими доменными объектами и их изменениями, обеспечивая четкие границы ответственности.

[plantuml]
----
@startuml
package "Catalog Service" {
    class Laptop
}

package "Shopping Cart Service" {
    class ShoppingCart
    class CartItem
}

package "Ordering Service" {
    class Order
    class OrderItem
}

package "Inventory Service" {
    class InventoryItem
    class StockReservation
}

package "Payment Service" {
    class Payment
}

package "Shipping Service" {
    class Shipment
}

Laptop --> CartItem
CartItem "*"-"1" ShoppingCart
ShoppingCart --> Order
OrderItem "*"-"1" Order
Order --> StockReservation
StockReservation --> Payment
StockReservation --> Shipment

InventoryItem - StockReservation

@enduml
----
