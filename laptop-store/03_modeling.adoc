= Анализ предметной области интернет-магазина с помощью Domain Storytelling и Event Storming

При проектировании интернет-магазина важно выбрать подходящий метод анализа предметной области. Два наиболее эффективных метода в контексте **Domain-Driven Design (DDD)** — это *Domain Storytelling* и *Event Storming*.

* **Domain Storytelling** помогает понять, как пользователи взаимодействуют с системой, и подходит для традиционной **синхронной архитектуры (REST API)**.
* **Event Storming** фокусируется на событиях и их взаимосвязях, что делает его идеальным для **событийно-ориентированной архитектуры (Kafka, RabbitMQ, CQRS, Event Sourcing)**.

В этом документе рассмотрим, как применить оба метода к анализу предметной области интернет-магазина ноутбуков и как на их основе выделить **API, DTO сервисов** и **событийную обработку заказа**.

== Domain Storytelling (Рассказывание историй домена)

=== Описание метода
Domain Storytelling — это визуальный метод, который позволяет анализировать процессы предметной области с помощью пошагового описания взаимодействий пользователей и системы. Он помогает:

- Определить ключевые роли (актеров) и их действия.
- Выявить основные шаги бизнес-процессов.
- Определить границы **bounded contexts**.
- Найти узкие места и точки интеграции.

=== Применение к интернет-магазину ноутбуков
Рассмотрим несколько пользовательских историй и отобразим их в виде **PlantUML-диаграммы**.

==== **Пользовательские истории**
1. **Просмотр каталога**
- Как клиент, я хочу просматривать список ноутбуков, чтобы выбрать подходящую модель.
- Как клиент, я хочу искать ноутбуки по характеристикам (размер экрана, RAM, SSD и т. д.), чтобы найти оптимальный вариант.
- Как клиент, я хочу видеть детали ноутбука (характеристики, цену, наличие), чтобы принять решение.

2. **Управление корзиной**
- Как клиент, я хочу добавить ноутбук в корзину, чтобы оформить заказ позже.
- Как клиент, я хочу изменять количество ноутбуков в корзине.
- Как клиент, я хочу удалить ноутбук из корзины, если передумал.

3. **Оформление заказа**
- Как клиент, я хочу оформить заказ, выбрав адрес и способ доставки.
- Как клиент, я хочу выбрать удобный способ оплаты.
- Как клиент, я хочу получить подтверждение, что заказ принят.

4. **Оплата заказа**
- Как клиент, я хочу оплатить заказ, чтобы его отправили в работу.
- Как клиент, я хочу увидеть статус оплаты, чтобы быть уверенным в её успешности.

==== **Диаграмма Domain Storytelling**

[plantuml]
----
@startuml
actor Клиент
participant "Каталог" as Catalog
participant "Корзина" as Cart
participant "Система заказов" as Ordering
participant "Склад" as Inventory
participant "Сервис оплаты" as Payment
participant "Сервис доставки" as Shipping
participant "Сервис отслеживания заказов" as Tracking

' --- Просмотр каталога ---
Клиент -> Catalog : Просматривает список ноутбуков
Клиент -> Catalog : Фильтрует по характеристикам
Клиент -> Catalog : Открывает страницу ноутбука

' --- Управление корзиной ---
Клиент -> Cart : Добавляет ноутбук в корзину
Клиент -> Cart : Изменяет количество ноутбуков
Клиент -> Cart : Удаляет ноутбук из корзины
Cart -> Catalog : Запрашивает актуальные данные о товаре

' --- Оформление заказа ---
Клиент -> Ordering : Оформляет заказ
Ordering -> Cart : Запрашивает содержимое корзины
Ordering -> Inventory : Проверяет наличие товара
Inventory -> Ordering : Подтверждает наличие
Ordering -> Payment : Запрашивает оплату

' --- Оплата заказа ---
Клиент -> Payment : Выполняет оплату
Payment -> Ordering : Подтверждает оплату
Ordering -> Shipping : Создаёт заявку на доставку
Shipping -> Клиент : Отправляет уведомление о доставке
Shipping -> Tracking : Обновляет статус заказа
Клиент -> Tracking : Проверяет статус заказа

@enduml
----

== Event Storming (Шторм событий)

=== Описание метода
Event Storming — это метод, который помогает анализировать предметную область через ключевые **события**. Он особенно полезен для **событийно-ориентированной архитектуры**, где микросервисы взаимодействуют асинхронно.

=== Как мы совместили Domain Storytelling и Event Storming
После того как мы выделили основные бизнес-процессы с помощью **Domain Storytelling**, мы перевели их в событийную модель **Event Storming**, чтобы определить границы сервисов, их API и DTO.

- **Каждое действие пользователя в Storytelling порождает одно или несколько событий в Event Storming.**
- **Мы выделили bounded contexts и связанные API.**
- **Создана событийная цепочка обработки заказа.**


==== **Обработка заказа на основе Event Storming**

=== Сравнение Domain Storytelling и Event Storming

Два метода анализа предметной области имеют свои особенности и применение:

[options="header"]
|===
| Критерий | Event Storming | Domain Storytelling

| **Фокус**
| События и реакция системы
| Пошаговые действия пользователей

| **Тип взаимодействия**
| Асинхронные события, pub/sub, message-driven
| Синхронные REST-запросы и ответы

| **Когда использовать?**
| Микросервисная, событийно-ориентированная архитектура
| Классическая архитектура, монолиты, REST API

| **Как строится диаграмма?**
| От событий (OrderPlaced, StockReserved)
| От шагов процесса (Клиент оформляет заказ)

| **Какие объекты ключевые?**
| События (Event), команды (Command), Saga
| Активные роли (пользователи, сервисы)

| **Обратная связь**
| Позволяет выявить гонки данных, дублирование, необходимость CQRS
| Помогает выявить узкие места в бизнес-процессах

| **Где используется?**
| Kafka, RabbitMQ, Event Sourcing, CQRS
| REST API, gRPC, традиционные веб-приложения
|===


== Заключение
* **Domain Storytelling** помог определить основные процессы и взаимодействия пользователей.
* **Event Storming** позволил разложить эти процессы на события, выделить bounded contexts и API.
* **Мы совместили оба метода**, получив как синхронные REST-вызовы, так и событийную модель обработки заказа.
